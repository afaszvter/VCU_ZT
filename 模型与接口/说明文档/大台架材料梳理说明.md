# 大台架材料梳理说明

## 1. 目标

本说明用于支撑两项工作：

1. 根据现有材料梳理模块间通信关系，形成 `DBC` 编写基线。
2. 根据线束/接口/协议材料搭建 `I/O Simulink` 模型骨架。

当前材料里同时存在：

- 整车级需求文档
- VCU 硬件手册
- VCU 线束接口梳理
- 整车 CAN 矩阵
- 电机/PDU/辅驱协议
- 三通阀协议
- 制动电阻协议与规格书
- 原理图图纸

这些文件并不完全一致，所以第一步不是直接写 DBC，而是先建立“基线版本”和“主数据来源”。

---

## 2. 建议的主数据来源

### 2.1 优先级结论

建议按下面顺序作为后续工作的“主来源”：

1. `VCU接口梳理 - 20260602.xlsx`
   用于新 VCU 管脚、I/O 类型、线束映射。
2. `整车CAN20260423.xls`
   用于最新整车 CAN/动力 CAN 报文矩阵。
3. `大台架VCU功能需求分析0421.docx`
   用于系统功能、模块关系、流程解释、缺失协议补充说明。
4. 各设备的专用协议文件
   用于补齐设备级细节，优先使用与 `整车CAN20260423.xls` 一致的版本。

   用于最终核对线束、继电器、接触器、供电与总线拓扑。

#### 作为主版本使用

### 2.3 已发现冲突

1. `ZT_SPV_CAN通讯协议_V1.1——电池电机.xls` 和 `开关磁阻电机MP2505_深蓝动力矿卡四合一通讯协议.xls` 中部分动力 CAN 报文 ID 不一致。
   例如驱动电机控制报文：
   - `ZT_SPV` 中 `VCU_DrvMCU1 = 0x0C31F0D0`
   - `整车CAN20260423` / `开关磁阻专用协议` 中对应为 `VCU2MCU2 = 0x0C31F1D0`

2. 三通阀存在两套协议来源：
   - `三通电磁阀2733523...xls`：250 kbps，ID 为 `0x0cfd6720 / 0x18FD20B0`
   - `Φ25 V560424A15三通比例阀can协议.xlsx`：500 kbps，ID 为 `0x18FF25EF / 0x18FF243B`
   其中 `整车CAN20260423.xls` 已采用后者，因此建议以后者为准。

3. `VCU接口梳理.xlsx` 是旧版接口表，没有新 VCU 管脚分配；`VCU接口梳理 - 20260602.xlsx` 已补充新 VCU 管脚，应以后者为准。

## 4. 目前能确定的模块和接口

## 4.1 控制器与节点

### 整车 CAN（建议命名 CANA）

- `VCU`
- `BMS`
- `OBC`
- `TMS`
- `ICU`
- `BRS/BRC` 制动电阻控制器
- 可能还有未使用的永磁同步 E3/E4、开关磁阻 18/19，但当前需求文档说明它们未使用

### 动力 CAN（建议命名 CANB）

- `VCU`
- `开关磁阻多合一 / MCU`
- `PDU / HVC`
- `DCDC`
- `DCAC(油泵)`
- `DCAC(气泵)`
- `三通比例阀 TWV`
- 主驱/辅驱电机控制器

### 调试 CAN（建议命名 CANC）

- 当前主要作为调试口，接口表中只有 VCU `CAN2H/CAN2L`

## 4.3 传感器/执行器硬件特征

### 电子油门踏板

来自 `b5f20a1102ca53c27e7d94f44ab56e32.jpg`：

- 双路供电、双路地、双路信号输出
- 典型 6 针
- 5V 供电
- 两路输出 `Vs1 / Vs2`
- 适合建成 `Pedal1Raw`、`Pedal2Raw`、`PedalPlausibility` 三块

### OKT7 温度变送器

来自 `OKT7温度变送器.pdf`：

- 输出型式可能为 `4...20mA`、`0...5V`、`0...10V`
- 供电 `12...30Vdc`
- 量程示例 `0...100°C`

结论：

- 必须先确认现场型号到底是电流型还是电压型。
- 若是 `4...20mA`，VCU 不能直接按普通 0~5V 模拟量处理，需确认前端是否有采样电阻或调理板。

### 制动电阻系统

来自 `制动电阻系统规格书（风冷100Kw）V7(1).pdf`：

- 控制电压 `DC24V`
- 高压输入/输出 `400~800VDC`
- 额定功率 `100kW`
- 与整车通讯 CAN 口：`CANAH / CANAL`
- 还带风扇控制、风扇故障、温度采集接口

结论：

- 你当前做 DBC 时，优先处理其 CAN 通信部分。
- 做 I/O 模型时，需要考虑它除了 CAN 以外还有接触器状态和可能的风扇/温度相关联动。

## 6. 做 DBC 时的建议

## 6.1 先分两个 DBC

不要先做一个大而全的 DBC，建议至少拆成两个：

1. `bench_vehicle_can.dbc`
   对应整车 CANA
2. `bench_power_can.dbc`
   对应动力 CANB

调试 CANC 可以先不建，除非现场有明确报文。

## 6.2 DBC 的编写顺序

### 第一步：先建节点

#### 整车 CANA 节点

- `VCU`
- `BMS`
- `OBC`
- `TMS`
- `ICU`
- `BRS`

#### 动力 CANB 节点

- `VCU`
- `PDU`
- `HVC`
- `MCU_Oil`
- `MCU_Drv`
- `DCDC`
- `ACU_OilPump`
- `ACU_AirPump`
- `TWV`

### 第二步：只录入消息头

先把每个报文的：

- 报文名
- ID
- DLC
- 周期
- 发送节点

全部建完，不要一边建报文一边抠每个 bit。

### 第三步：补信号

优先补下面几类信号：

1. 状态机必需信号
   如上下电、继电器状态、Ready、故障等级、唤醒
2. 控制闭环必需信号
   如扭矩、转速、风扇档位、功率请求
3. 监控显示信号
   如温度、电压、电流、SOC、故障码

### 第四步：补枚举和无效值

尤其要补：

- 故障等级
- 工作模式
- 继电器状态
- 档位/控制模式
- 心跳计数

## 6.3 需要优先确认的冲突项

1. 动力电机到底采用：
   - `开关磁阻专用协议`
   - 还是 `ZT_SPV` 中那套 ID

2. 三通阀现场实物到底是：
   - `Φ25 V560424A15`
   - 还是 `2733523`

3. 制动电阻接触器状态是否由高压配电柜报文提供，还是由别的控制器提供。

4. 永磁同步与开关磁阻是否同时保留在 DBC 内，还是只保留当前台架用的一套。

## 7. 做 Simulink I/O 模型时的建议

## 7.1 推荐模型分层

建议拆成 5 层：

1. `HAL_IO`
   物理量采集与驱动，直接对应 AI/DI/PWM/LSO/HSO/CAN 驱动
2. `IF_CAN_RX`
   各路 CAN 报文解包
3. `IF_CAN_TX`
   各路 CAN 报文打包
4. `SIG_COND`
   量纲换算、滤波、故障诊断、双通道一致性检查
5. `APP_CTRL`
   上下电、充电、冷却、驱动、附件控制逻辑

## 7.2 先搭骨架，不要先写逻辑

第一轮模型先做到：

- 所有输入口有名字
- 所有输出口有名字
- 所有 CAN 报文都有收发容器
- 所有重要信号都有 Bus/Struct 对应

不要一开始就进入状态机细节。

## 7.3 建议先建的 Simulink 子系统

1. `VCU_InputIf`
   - Key
   - Gear
   - Pedal
   - Switch
   - Temp

2. `VCU_OutputIf`
   - Pump
   - Fan1PWM
   - Fan2PWM
   - Horn
   - Relay
   - Valve outputs

3. `CANA_Rx`
   - BMS
   - OBC
   - TMS
   - BRS

4. `CANA_Tx`
   - VCU_BMS
   - VCU2ICU
   - VCU2BRS
   - VCU2TMS

5. `CANB_Rx`
   - PDU/HVC
   - MCU
   - DCDC/DCAC
   - TWV

6. `CANB_Tx`
   - VCU_PDU
   - VCU2MCU
   - VCU_DCDC1
   - Ctrl_HVC
   - VCU_ACU1
   - VCU_TWV

7. `PowerStateMgr`
   - 低压上电
   - 高压上电
   - 常规下电
   - 异常下电

8. `ChargeMgr`
   - 慢充请求
   - BMS 交互
   - OBC 唤醒

9. `CoolingMgr`
   - TMS 档位请求
   - 双风扇 PWM
   - 水泵控制

10. `ActuatorMgr`
   - 阀类
   - 喇叭
   - 缓速器

## 7.4 量纲换算建议

应把以下内容单独做成换算模块，不要散落在业务逻辑里：

- 油门双通道比例换算
- 温度变送器量程换算
- PWM 百分比与占空比换算
- CAN 物理量 `raw <-> physical` 换算
- 故障码/枚举值解释

## 8. 你接下来最应该怎么做

## 8.1 第一阶段：冻结基线

输出物：

- 一份 `协议采用清单`
- 一份 `冲突项清单`

动作：

1. 认定 `VCU接口梳理 - 20260602.xlsx` 为 I/O 基线。
2. 认定 `整车CAN20260423.xls` 为总线基线。
3. 对动力系统冲突项做一次人工确认。
4. 对三通阀型号做一次人工确认。

## 8.2 第二阶段：先做信号总表

输出物：

- `Signal List` 总表，建议列：
  - `Network`
  - `Node`
  - `Direction`
  - `Message`
  - `ID`
  - `Signal`
  - `Unit`
  - `Scale`
  - `Offset`
  - `Type`
  - `Source File`
  - `Status`

动作：

1. 先录整车 CAN。
2. 再录动力 CAN。
3. 再录物理 I/O。

## 8.3 第三阶段：先出 DBC 框架

输出物：

- `bench_vehicle_can.dbc`
- `bench_power_can.dbc`

动作：

1. 先建节点和报文头。
2. 再补关键控制信号。
3. 最后补诊断和显示信号。

## 8.4 第四阶段：搭 Simulink I/O 骨架

输出物：

- `bench_io_top.slx`
- `cana_if.slx`
- `canb_if.slx`
- `io_hal.slx`

动作：

1. 先搭端口和总线。
2. 再接报文打包解包。
3. 最后挂控制逻辑。

## 8.5 第五阶段：做联调检查

检查项：

1. 报文 ID 是否与线束接入总线一致。
2. 波特率是否一致。
3. Intel/Motorola 是否一致。
4. 信号方向是否一致。
5. 物理量范围是否合理。
6. 心跳/超时/默认值是否定义。

## 10. 建议你立刻开始的顺序

如果现在就开始推进，建议顺序固定为：

1. 先把 `VCU接口梳理 - 20260602.xlsx` 整理成一张 `I/O 总表`
2. 再把 `整车CAN20260423.xls` 整理成 `整车 CAN 信号表`
3. 再把 `开关磁阻电机MP2505...xls` + `PDU...xlsx` + `三通阀...xlsx` 整理成 `动力 CAN 信号表`

5. 再搭 Simulink 的 `HAL_IO + CANA_RX/TX + CANB_RX/TX`
6. 最后再接 `高压状态机 / 慢充状态机 / 冷却控制`
